# v0.4.0 B1 Implementation Plan

> Native Messaging Auto-Link: remove manual token provisioning,
> replace with `browser.runtime.connectNative()` auto-launch.

**Status**: Draft — pending prototype validation  
**RFC**: [RFC-B1-NM-Auto-Link.md](rfcs/RFC-B1-NM-Auto-Link.md)  
**Spike**: [spikes/b1-ipc-spike/](../../spikes/b1-ipc-spike/)

---

## 1. B1 Goal

Currently (v0.3.4), the user must manually:

1. Start the bridge (`cargo run`)
2. Copy the auto-generated token from a file
3. Paste it into the Extension Options page

B1 replaces this with a transparent flow:

1. The extension calls `browser.runtime.connectNative("org.brp.bridge")`
2. Firefox spawns the bridge as a child process
3. The bridge sends the token via **stdout** (Native Messaging format) — no file, no clipboard
4. The extension reads the token from the `connectNative` port, connects to the bridge's WebSocket
5. The entire flow completes without any user interaction

### Native Messaging format

```
[uint32 LE: message length][length bytes: UTF-8 JSON]
```

This is Firefox's standard Native Messaging wire format. Both bridge (stdin read + stdout write) and extension (`port.postMessage` / `port.onMessage`) use it.

## 2. Sub-PR Breakdown (First Batch — 6 PRs)

### PR #18: Bridge Dual Mode (`--mode=bridge` / `--mode=bootstrap`)

**Scope**:
- Add `--mode` CLI flag to the bridge binary:
  - `bridge` (default): WebSocket server + Native Messaging I/O — existing behavior, unchanged
  - `bootstrap` (new): minimal binary launched by Firefox via `connectNative` — only does token delivery via stdout, then hangs as a keepalive. Does NOT start a WebSocket server, does NOT read stdin beyond the first handshake.
- Add `--echo` smoke test flag: when set, the binary echoes back any JSON received on stdin (diagnostic mode, not used in production).

**Dependencies**: None.  
**Risk**: Low — adds flags to existing binary, no new IPC.  
**Acceptance**:
- `cargo run -- --mode=bootstrap --echo` → `echo -ne '<length>{"test":1}\n' | cargo run -- --mode=bootstrap --echo` → echoes back `{"test":1}` on stdout in NM format
- Existing `--mode=bridge` (default) behavior unchanged

---

### PR #19: Unix Socket IPC (Linux/macOS)

**Scope**:
- Bootstrap process creates a Unix Domain Socket at a well-known path (`$XDG_RUNTIME_DIR/brp-bridge.sock` or fallback)
- Socket used ONLY for lockfile coordination (single-instance enforcement) — NOT for message passing or instance reuse
- Existing bridge checks the socket before starting its WebSocket server

**Dependencies**: PR #18 (dual-mode binary)  
**Risk**: Medium — Unix sockets are straightforward on Linux, macOS has slight path differences.  
**Acceptance**:
- Two bootstrap processes cannot bind the same socket (second one exits with clear error)
- Exiting bridge cleanly removes the socket file
- Stale socket from a crash is detected and overwritten

---

### PR #20: Windows Named Pipe (with DACL)

**Scope**:
- Bootstrap process creates a Named Pipe at `\\.\pipe\brp-bridge`
- Pipe used for lockfile coordination (same pattern as Unix Socket)
- **DACL**: Construct a `SECURITY_ATTRIBUTES` with a DACL that grants access only to the current user SID. This prevents other users on the same machine from connecting. (Deferred from v0.3.2 spike — the spike verified the approach but did not implement DACL.)
- `CreateNamedPipeW` with `PIPE_ACCESS_DUPLEX`, `FILE_FLAG_FIRST_PIPE_INSTANCE`

**Dependencies**: PR #18  
**Risk**: **High** — DACL/SECURITY_ATTRIBUTES/SID API in Windows is complex and poorly documented. User has Windows test machine available for verification.  
**Acceptance**:
- Bootstrap creates named pipe successfully
- Second bootstrap process fails because `FILE_FLAG_FIRST_PIPE_INSTANCE` is already taken
- Non-owner user processes cannot connect (verified by trying from a different user account)
- DACL is constructed without unsafe pointer casts or UB

---

### PR #21: PID Lockfile + Stale Cleanup

**Scope**:
- Bridge writes a PID lockfile on startup: `{rundir}/brp-bridge.lock` containing `{"pid": <pid>, "port": <port>}`
- On startup, check for existing lockfile:
  - If PID is alive (checked via `kill(pid, 0)` on Unix / `OpenProcess` + `GetExitCodeProcess` on Windows → port is valid), refuse to start "Bridge already running"
  - If PID is dead, clean up stale lockfile and start fresh
- **No MRU multi-instance reuse**: each extension instance starts its own bridge on its own port. If two Firefox windows are open, they get independent bridges. MRU reuse is deferred to v0.4.2.
- Port assignment: bridge picks a random available port (bind to port 0, read back the assigned port)
- The assigned port is written to the lockfile AND sent to the extension via bootstrap token delivery (so the extension knows where to connect)

**Dependencies**: PR #19, PR #20  
**Risk**: Medium — PID reuse is the main edge case. When a PID is reused by the OS, a stale lockfile with a live PID would block startup. Mitigation: additionally check process creation time or command line.  
**Acceptance**:
- First bridge starts and writes lockfile
- Second bridge (same user) refuses: "Bridge already running (PID XXXX, port YYYY)"
- Kill first bridge → second bridge detects stale PID, cleans lockfile, starts
- PID reuse edge case documented (not solved — out of scope for v0.4.0)

---

### PR #22: Extension connectNative + Auto-Link

**Scope**:
- Extension adds `nativeMessaging` permission to `manifest.json`
- New `src/native.ts` module: `startBridge()` function
  1. Calls `browser.runtime.connectNative("org.brp.bridge")`
  2. Reads the first message from the port (token + port number)
  3. Closes the `connectNative` port
  4. Connects to the bridge WebSocket at `ws://127.0.0.1:<port>` with the received token
  5. Falls back gracefully if `connectNative` fails (bridge not installed) → shows a user-friendly error
- Update `background.ts`:
  - On startup, try `startBridge()` first
  - If successful, skip manual token lookup from `browser.storage.local`
  - If `connectNative` not available (e.g., extension loaded via `about:debugging` without native manifest), fall back to manual WS connection (existing behavior)

**Dependencies**: PR #18 (bootstrap mode), PR #21 (port assignment in lockfile → token message)  
**Risk**: Medium — Firefox's `connectNative` behavior has edge cases (see prototype validation below).  
**Acceptance**:
- Load extension → bridge auto-starts → WebSocket connected → initialize handshake succeeds
- No token file needed, no user interaction
- **Token timeout (from 5d prototype finding)**: Extension connects via `connectNative`, starts a 3-second timer. If no token message received within 3s, treat as "bridge not installed", show user-friendly error: "Bridge not installed. Run install.sh (Linux/macOS) or install.ps1 (Windows) first."
  - *Rationale*: Prototype 5d confirmed Firefox fires `onDisconnect` but `browser.runtime.lastError` has no useful info when native manifest path is wrong. Extension must actively detect this case.
- Bridge process exits when Firefox closes

## 4. Risk Register

| Risk | Severity | Mitigation |
|------|:---:|------------|
| Windows DACL API complexity | High | Spike (`spikes/b1-ipc-spike/`) has working reference code; PR #20 built on spike |
| Extension lifecycle + subprocess coupling | Medium | Bootstrap process keeps extension port alive; on disconnect, bridge exits |
| PID reuse (lockfile stale detection false positive) | Medium | In addition to PID check, verify process creation time; document remaining edge case |
| Lockfile corruption (crash mid-write) | Low | Atomic write (write to .tmp, fsync, rename) |
| Multi-user (lockfile path collisions) | Low | Lockfile in user-private directory (`$XDG_RUNTIME_DIR` / `$LOCALAPPDATA`), not `/tmp` |
| Firewall / antivirus blocking WebSocket on random port | Low | Loopback-only; most AV allows loopback |

## 5. Boundary Cases (Per-PR Acceptance Checklist)

Each PR must verify these before merge:

| Case | Expected behavior | Verified in PR |
|------|-------------------|:---:|
| Extension unloaded (`about:debugging` → Remove) | Bridge receives process termination signal, exits cleanly | |
| Browser crashes | Bridge detects port disconnect, exits within 30s timeout | |
| User closes browser normally | Bridge exits cleanly (port disconnect → cleanup → exit) | |
| Bridge crashes | Extension detects WebSocket close, shows error, offers retry | |
| Bridge startup fails (port occupied) | Bridge logs error, sends error to extension via bootstrap port, exits | |
| NM message format error | Bridge logs warning, skips malformed message, does not crash | |
| Native manifest not installed | Extension connects, 3s token timeout fires, shows "bridge not installed" error with install instructions | |
| Two bridges started concurrently | Lockfile detects first instance, second exits with clear message | |

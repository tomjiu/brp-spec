### PR #27: E3 DOM Precondition Validation (~2-3 days)

**Problem**: AI says "click login button", extension finds `.btn` and clicks, but may click wrong element (footer "login" link instead of actual button).

**Solution**: AI can attach `precondition` to element actions. Extension validates element before acting, rejects if mismatch.

**Scope**:

**Rust side** (`bridge/src/protocol/`):
- New `Precondition` type in wire types:
  ```rust
  #[derive(Debug, Clone, Serialize, Deserialize, TS)]
  #[serde(rename_all = "camelCase")]
  #[ts(rename_all = "camelCase", export, export_to = "../bindings/")]
  pub struct Precondition {
      pub tag_name: Option<String>,        // "BUTTON", "A", "INPUT"
      pub text_contains: Option<String>,   // element text must include this
      pub attributes: Option<HashMap<String, String>>,
  }
  ```
- `element.click` / `element.fill` / `element.type` / `element.hover` / `element.select` params add optional `precondition` field
- ts-rs regenerate bindings

**TS side** (`extension/src/types.content.ts` + `content.ts`):
- `ContentMessage` add optional `precondition?: Precondition`
- `content.ts` action handlers receive `precondition`, validate element before acting:
  ```typescript
  function validatePrecondition(el: Element, pre?: Precondition): boolean {
    if (pre?.tagName && el.tagName.toUpperCase() !== pre.tagName.toUpperCase()) return false;
    if (pre?.textContains && !el.textContent?.includes(pre.textContains)) return false;
    if (pre?.attributes) {
      for (const [k, v] of Object.entries(pre.attributes)) {
        if (el.getAttribute(k) !== v) return false;
      }
    }
    return true;
  }
  ```
- Validation failure returns new error `BRP_PRECONDITION_FAILED` with actual element info (tagName, text, attributes)

**Dependencies**: PR #25 (TS baseline)
**Risk**: Medium — wire type change, need ts-rs regen + TS type sync
**Acceptance**:
- AI sends `element.click` with `precondition: {tagName: "BUTTON", textContains: "登录"}`
- Finds `.btn` but it's `<a>` → reject, return `BRP_PRECONDITION_FAILED` with actual element info
- Finds `<button>登录</button>` → execute click
- Rust unit tests: Precondition serde round-trip
- TS unit tests: validatePrecondition all combinations
- Integration test: AI sends with precondition, verify reject on mismatch

### PR #28: E4 Context Recovery Pipeline (~2-3 days, serial after E3)

**Problem**: AI sends CSS selector `.login-btn`, but page class changed (now `.login-button`), selector fails, AI stuck.

**Solution**: When selector fails, automatically retry via fallback chain: `nodeId → role → css → xpath → coordinate → text`.

**Scope**:

**Rust side**:
- `Selector` type already has `type` field
- New `acceptFallback: bool` field in element action params
- When `acceptFallback=true` and main selector fails, bridge notifies extension to try other selector types

**TS side** (`extension/src/content.ts`):
- `findElement` refactor: main selector fails + `acceptFallback=true` → try fallback chain
  ```typescript
  async function findElementWithFallback(
    selector: SelectorValue,
    acceptFallback: boolean
  ): Promise<{ element: Element | null; matchedType?: SelectorType }> {
    // 1. Try main selector
    let el = findBySelector(selector);
    if (el) return { element: el, matchedType: selector.type };
    if (!acceptFallback) return { element: null };
    
    // 2. Fallback chain
    const fallbacks: SelectorType[] = ["nodeId", "role", "css", "xpath", "coordinate", "text"];
    for (const type of fallbacks) {
      if (type === selector.type) continue;
      el = findBySelector({ ...selector, type });
      if (el) {
        console.log(`[BRP] Selector fallback: ${selector.type} → ${type}`);
        return { element: el, matchedType: type };
      }
    }
    return { element: null };
  }
  ```
- Action result includes `matchedSelector: {type}` to tell AI which selector actually matched

**Dependencies**: PR #27 (E3 establishes the validation pattern E4 builds on)
**Risk**: Medium — fallback chain may have subtle ordering issues, needs thorough testing
**Acceptance**:
- AI sends `element.click` with `selector: {type: "css", value: ".old-btn"}` + `acceptFallback: true`
- `.old-btn` doesn't exist → try text match "登录" → find → click
- Returns `matchedSelector: {type: "text"}`
- TS unit tests: various fallback scenarios
- Integration test: main selector fails, fallback to text succeeds

## 3. After v0.4.1

### v0.4.2 — Real Extension Integration Tests (mandatory)

After B1 + E3/E4, the call chain becomes complex:
```
AI → Bridge → Background → Content Script → DOM
```

Unit tests can't cover this. v0.4.2 must add:
- web-ext + headless Firefox
- End-to-end test coverage: connect → navigate → find element → click → fill → reconnect
- Fixed test pages (local SPA) for reliable selectors
- Retry mechanism for flaky tests

**Why mandatory**: Without integration tests, future features will cause regressions that CI can't catch. The race conditions found in B1 (port.disconnect vs stdin EOF, token not stored) were all integration issues.

### v0.5.0+ — Defense in Depth (after stability)

- E1 Permission gating (confirm dialogs for navigation, form submit, script.execute)
- E2 Domain blacklist (user-configurable, wildcard support)
- E5 Screenshot permission gate, log sanitization, CSP hardening
- B2 Multi-token (per-client tokens, independent revocation)

### v1.0-pre — Stable API + Compat

- supportsAutoLink capability negotiation
- Backward compatibility test matrix
- Multi-browser compatibility (Firefox / Zen / LibreWolf)

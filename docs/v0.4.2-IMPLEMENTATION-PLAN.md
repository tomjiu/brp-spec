# v0.4.2 Implementation Plan

> Integration Testing + Conditional MRU

**Status**: Draft
**Prerequisite**: v0.4.1 released (tag `v0.4.1`)
**Companion**: [`docs/v0.4.1-IMPLEMENTATION-PLAN.md`](v0.4.1-IMPLEMENTATION-PLAN.md)

---

## 1. v0.4.2 Goal

v0.4.0 (B1) + v0.4.1 (E3/E4) made the call chain complex:
```
AI → Bridge → Background → Content Script → DOM
```

Unit tests can't cover integration paths. Race conditions found in B1 (port.disconnect vs stdin EOF, token not stored) were all integration issues. v0.4.2 establishes integration test baseline.

### Non-goals (deferred)

| Item | Deferred to | Reason |
|------|:---:|--------|
| Defense in Depth (permission gating) | v0.5.0 | Stability first |
| supportsAutoLink capability | v1.0-pre | Not needed until multiple versions |

## 2. Sub-PR Breakdown (6-7 PRs)

### PR #33: Phase 1 — Bridge Standalone Smoke Test (1-2 days)

**Scope (revised)**: Bridge standalone mode smoke test only.
B1 auto-link integration test deferred to Phase 2 (requires xvfb).

**Why revised**: Firefox headless mode does not officially support native messaging
(connectNative). Mozilla docs don't guarantee this works. Rather than fight
headless incompatibility:
- Phase 1: verify bridge startup + WS + JSON-RPC (standalone mode, fixed port)
- Phase 2: use xvfb-run firefox (non-headless) for B1 auto-link full chain test

**Scope**:
- New `tests/integration/` directory
- Bridge startup (standalone mode, port 9817)
- WS connection establishment
- JSON-RPC initialize handshake
- `browser.list` basic response
- CI: integration job (ubuntu-only, PR-triggered)

**Not in scope (deferred to Phase 2)**:
- B1 auto-link via connectNative
- Extension loading in headless Firefox
- Bridge auto-start from extension

**Acceptance**:
- Bridge process starts successfully
- WS connection established
- JSON-RPC initialize handshake succeeds
- CI ubuntu integration job green

---

### PR #34: Phase 2 — Bridge Protocol Contract + Extension Logic (2-3 days)

**Scope (revised)**: Split into Phase 2a (bridge protocol contract) + Phase 2b
(extension logic). B1 auto-link deferred to manual testing.

**Why revised**: Spike PR #34 confirmed Firefox CI environment forces headless mode
even with xvfb (`*** You are running in headless mode.`). Firef ox auto-detects
CI/container env and enters headless regardless of virtual display. In headless
mode, `connectNative()` does not work — B1 auto-link cannot be CI-tested.

**Phase 2a: Bridge Protocol Contract (CI)**:
- stdin/stdout + mock extension
- initialize / browser.list / tab.list / element actions
- Error handling (no extension connected, invalid methods)
- E3/E4 bridge-side processing (if applicable)

**Phase 2b: Extension Logic (CI)**:
- Playwright Firefox headless + mock bridge
- Extension loading (`--load-extension`)
- E3 precondition validation (content script)
- E4 fallback chain (content script)
- No connectNative needed — headless → OK

**B1 auto-link (manual only)**:
- Verified on real Windows machine (PR #23 5b → v0.4.0)
- Pre-release manual checklist item
- CI does not cover this path

**Acceptance**:
- Bridge protocol contract tests pass (CI)
- Extension logic tests pass (CI)
- B1 auto-link manually verified on real machine (pre-release)

### PR #36: (Conditional) MRU Multi-Instance Reuse (2-3 days)

**Prerequisite**: Phase 1-3 complete. Measure reconnect time.

| Reconnect time | Decision |
|---|---|
| < 1s | MRU low value, skip |
| > 3s | MRU worth doing |

**Scope (if done)**:
- `ipc_unix.rs` + `ipc_windows.rs`: MRU selection logic
- `lockfile.rs`: multi-instance registration
- `native.ts`: query available bridge before starting new one
- New `supportsMRU` capability

**Acceptance**:
- Reconnect time drops from Xs to < 500ms
- Multiple extension instances share one bridge

## 3. Tool Decision

**Playwright** chosen for integration testing:

- Modern API, clean
- Good Firefox headless support
- Active community
- Alternative: web-ext + selenium (heavier), raw `firefox --headless` (lighter but more manual)

---

## 4. Risk Register

| Risk | Severity | Mitigation |
|------|:---:|------------|
| Headless Firefox native messaging support | High | Phase 1 smoke test validates early; fallback to non-headless if needed |
| CI Firefox install/version drift | Medium | Pin Firefox version in CI |
| Integration test flaky on CI | Medium | Retry mechanism, generous timeouts |
| Playwright Firefox extension loading | Medium | Use `--load-extension` flag or temporary profile |
| MRU adds complexity without clear benefit | Low | Conditional on Phase 1-3 baseline data |

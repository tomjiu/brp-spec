# v0.5.2 Implementation Plan

> Tab-level Permissions + Browser History Access

**Status**: Draft
**Prerequisite**: v0.5.1 released (tag `v0.5.1`)

## 1. v0.5.2 Goal

v0.5.1 gave users "visibility" (icon + indicator + allowlist). v0.5.2 gives users "control":
- User decides which tabs AI can operate (tab-level permissions)
- User can optionally grant browser history access
- Failed permission prompts auto-revoke tab control

### Non-goals (deferred)

| Item | Deferred to | Reason |
|------|:---:|--------|
| tabGroups coloring | v0.6.0 | Zen compatibility unknown, needs real-machine test |
| Persistent controllable state across restart | v0.6.0 | Session-level is safer, use domain allowlist for long-term trust |

## 2. Core Design

### 2.1 Tab Controllable State

| Tab Source | Default State | User Action |
|---|---|---|
| AI opens via `tab.open` | Controllable | Can demote |
| AI navigates via `page.navigate` | Controllable | Can demote |
| User opens manually | Not controllable | Can promote |

### 2.2 Check Order

```
AI request →
  1. Tab controllable check → reject BRP_TAB_NOT_CONTROLLABLE if not
  2. v0.5.1 Allowlist → skip E2+E1 if matched
  3. E2 Blacklist → hard block
  4. E1 Permission gate → ask
  5. execute
```

### 2.3 Auto-Demotion Trigger

Tab auto-demotes to "not controllable" ONLY when:
- User clicks "Deny" on E1 permission dialog (BRP_PERMISSION_DENIED)

Does NOT demote on:
- E2 blacklist block (safety, not tab's fault)
- Normal errors (element not found, timeout)
- Bridge internal errors
